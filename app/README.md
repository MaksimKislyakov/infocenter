# 📋 Бизнес-логика системы прав: полное руководство

---

## 🔹 1. Модели и за что они отвечают

| Модель | Зачем нужна | Ключевые поля |
|--------|-------------|--------------|
| **`Unit`** | Организационная структура: дерево «Предприятие → Цех → Участок» | `id`, `name`, `level_type` (enterprise/shop/area), `parent_id` |
| **`User`** | Пользователи системы | `id`, `login`, `is_global_admin` |
| **`UserUnitPermission`** | **Ядро системы**: связь «кто → где → что может» | `user_id`, `unit_id`, `block`, `action` |
| **`Diagram`** | Диаграммы, к которым даётся доступ | `id`, `block`, `unit_id`, `columns`, `rows` |

### 🎯 Ключевая идея
> **Одно право = одна запись в `UserUnitPermission`**  
> Пример: «Иван может **просматривать** диаграммы блока **Безопасность** в **Цехе №1**»

```
{
  "user_id": "uuid-ивана",
  "unit_id": "uuid-цеха-1",
  "block": "safety",
  "action": "view"
}
```

---

## 🔹 2. Типы действий (`action`) и блоков (`block`)

### Действия
| Значение | Что разрешает |
|----------|--------------|
| `view` | Просматривать диаграммы |
| `manage` | Создавать/редактировать/удалять диаграммы |
| `manage_permissions` | Выдавать права другим (только админам) |

### Блоки (домены инфоцентра)
| Значение | Описание |
|----------|----------|
| `safety` | Безопасность |
| `quality` | Качество |
| `production` | Производство |
| `costs` | Затраты |
| `culture` | Корпоративная культура |
| `all` | Все блоки сразу (удобно для админов) |

## 🔹 3. Иерархия и наследование прав

```
Предприятие (enterprise)
├── Цех №1 (shop)
│   ├── Участок сборки (area)
│   └── Участок контроля (area)
└── Цех №2 (shop)
    └── Участок логистики (area)
```

### 🔄 Как работает наследование
| Право выдано на | Автоматически даёт доступ к |
|----------------|---------------------------|
| Предприятию | Всем цехам и участкам внутри |
| Цеху | Всем участкам внутри этого цеха |
| Участку | Только этому участку |

**Пример:**  
Если у пользователя есть `view` на `Цех №1` + `block: safety`, он увидит все диаграммы безопасности во всех участках этого цеха.

## 🔹 4. Как фронту общаться с бэком: эндпоинты

### 🏗️ Настройка структуры (делает админ, единоразово)

| Метод | Эндпоинт | Зачем | Пример запроса |
|-------|----------|-------|---------------|
| `POST` | `/units` | Создать подразделение | `{"name": "Цех №5", "level_type": "shop", "parent_id": "uuid-предприятия"}` |
| `GET` | `/units/tree` | Получить дерево для чекбоксов | — |

**Ответ `/units/tree`:**
```json
[
  {
    "id": "uuid-предприятия",
    "name": "Предприятие 999",
    "level": "enterprise",
    "children": [
      {
        "id": "uuid-цеха",
        "name": "Цех 999",
        "level": "shop",
        "children": [
          {"id": "uuid-участка", "name": "Участок 999", "level": "area"}
        ]
      }
    ]
  }
]
```

---

### 👥 Управление правами пользователей (админ)

| Метод | Эндпоинт | Зачем | Пример запроса |
|-------|----------|-------|---------------|
| `POST` | `/permissions/users/{user_id}` | Выдать права | См. ниже ↓ |
| `GET` | `/permissions/users/{user_id}` | Посмотреть права пользователя | — |
| `DELETE` | `/permissions/users/{user_id}` | Отозвать права | `{"permissions": [...]}` |

**Выдача прав (массив «галочек»):**
```json
POST /permissions/users/uuid-пользователя
{
  "permissions": [
    {
      "unit_id": "uuid-цеха",
      "block": "safety",
      "action": "view"
    },
    {
      "unit_id": "uuid-цеха", 
      "block": "quality",
      "action": "manage"
    },
    {
      "unit_id": "uuid-участка",
      "block": "production",
      "action": "view"
    }
  ]
}
```

**Ответ:**
```json
[
  {
    "id": "uuid-права",
    "unit": {
      "id": "uuid-цеха",
      "name": "Цех 999",
      "level_type": "shop"
    },
    "block": "safety",
    "action": "view"
  }
]
```

### 📊 Работа с диаграммами (обычный пользователь)

| Метод | Эндпоинт | Зачем | Тело запроса |
|-------|----------|-------|-------------|
| `GET` | `/diagrams?block=safety&unit_id=uuid` | Получить список диаграмм | — |
| `GET` | `/diagrams/{diagram_id}` | Получить одну диаграмму | — |
| `POST` | `/diagrams` | Создать диаграмму | См. ниже ↓ |
| `PUT` | `/diagrams/{id}` | Обновить диаграмму | См. ниже ↓ |
| `DELETE` | `/diagrams/{id}` | Удалить диаграмму | — |

**Создание диаграммы:**
```json
POST /diagrams
{
  "diagramm_id": "diag-123",
  "block": "safety",           // к какому блоку относится
  "unit_id": "uuid-участка",   // к какому подразделению привязана
  "columns": [
    {"name": "incident_type", "type": "string"},
    {"name": "count", "type": "number"}
  ],
  "rows": [
    {"incident_type": "Пожар", "count": 3},
    {"incident_type": "Травма", "count": 1}
  ]
}
```

## 🔹 6. Что отправлять с фронта: шпаргалка

### 📦 При создании пользователя + выдаче прав
```javascript
// 1. Создать пользователя
await api.post('/users', {
  login: 'ivanov',
  password: '***',
  full_name: 'Иванов И.И.'
});

// 2. Получить дерево подразделений
const tree = await api.get('/permissions/units/tree');

// 3. Собрать выбранные галочки (пример структуры на фронте)
const selectedPermissions = [
  { unit_id: 'uuid-цеха', block: 'safety', action: 'view' },
  { unit_id: 'uuid-цеха', block: 'quality', action: 'manage' }
];

// 4. Отправить права
await api.post(`/permissions/users/${userId}`, {
  permissions: selectedPermissions
});
```

### 📦 При создании диаграммы
```javascript
// Фронт уже знает: пользователь в "Цех №1", открыл блок "Безопасность"
await api.post('/diagrams', {
  diagramm_id: generateId(), // или сгенерировать на беке
  block: 'safety',           // из текущего контекста
  unit_id: 'uuid-цеха',      // из текущего контекста
  columns: [...],
  rows: [...]
});
```

### 📦 При загрузке списка диаграмм
```javascript
// Фронт передаёт фильтры, бэк сам отфильтрует по правам
const diagrams = await api.get('/diagrams', {
  params: {
    block: 'safety',         // текущий блок
    unit_id: 'uuid-цеха'     // текущее подразделение (опционально)
  }
});
// Если у пользователя нет прав — получит пустой массив или 403
```

## 🔹 7. Обработка ошибок: что показывать пользователю

| Статус | Когда возвращается | Что показать на фронте |
|--------|-------------------|----------------------|
| `403` | Нет права на действие/блок/юнит | «У вас нет доступа к этому разделу» + кнопка «Запросить доступ» |
| `404` | Диаграмма или подразделение не найдено | «Элемент не найден» |
| `400` | Некорректные данные в запросе | «Проверьте заполненные поля» + подсветка ошибок |

## 🔹 8. Чеклист для фронтенд-разработчика

- [ ] При загрузке админки пользователя: `GET /permissions/units/tree` → построить nested checkbox-дерево
- [ ] При сохранении прав: собрать массив `{unit_id, block, action}` → `POST /permissions/users/{id}`
- [ ] При работе обычного пользователя: всегда передавать `block` и `unit_id` в запросах к `/diagrams`
- [ ] Ловить `403` и показывать понятное сообщение, а не «ошибка сервера»
- [ ] Кэшировать дерево подразделений (оно редко меняется)

## 🔹 10. Если что-то не работает: диагностика

```bash
# 1. Проверить, есть ли права у пользователя
GET /permissions/users/{user_id}

# 2. Проверить, привязана ли диаграмма к правильному unit/block
GET /diagrams/{id}

# 3. Проверить иерархию: кто родитель участка?
# (нужно добавить эндпоинт, если нет)
GET /units/{unit_id}

# 4. Посмотреть логи бэка: там будет точная причина 403
docker compose logs -f backend | grep "Access denied"
```

✅ **Итого:**  
- Админ управляет структурой и правами через `/units` и `/permissions`  
- Обычный пользователь работает только с `/diagrams`, бэк сам фильтрует по его правам  
- Наследование работает автоматически: право на цех = право на все участки внутри  
- Фронт всегда передаёт `block` и `unit_id` в запросах к диаграммам
